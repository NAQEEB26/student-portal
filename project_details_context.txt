9. Authentication & Authorization 
Firebase 
ď‚· Use Firebase Auth (Email/password). Store roles in a users collection or in custom claims (for 
server trust). For role-based access use Firebase Security Rules or Cloud Functions to set 
custom claims. 
Supabase 
ď‚· Use Supabase Auth for registration/login. Store role and campus_id in a separate users table. 
Use Row Level Security (RLS) policies to restrict access: campus managers can only CRUD 
records where campus_id = their campus. 
10. Security Rules / Policies (Examples) 
Firestore rules (high level) 
ď‚· Only authenticated users read/write. 
ď‚· Admins can access all documents. 
ď‚· Campus managers can read/write students where resource.data.campusId == 
request.auth.uid_campusId (or compare to manager's assigned campus). 
Supabase RLS rules (concept) 
ď‚· Enable RLS on students table. 
ď‚· Policy for campus managers: allow select, update, delete where auth.role = 'campus_manager' AND 
campus_id = auth.campus_id. 
ď‚· Students can select their own record only.

15. Sample Workflows 
A. Add Student (Admin) 
1. Admin -> Students -> Add Student 
2. Fill form (including campus selection) 
3. Save -> student document inserted in DB -> upload photo to Storage 
4. Optionally send welcome email (via Cloud Function or server) 
B. Campus Manager Deletes Student 
1. Campus manager navigates to Students filtered by their campus 
2. Click delete on a student 
3. System checks policy (manager's campus matches student's campus) 
4. Delete student record and related storage (photo) and log the action
